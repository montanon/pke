# Security and Abuse Reporting

## Purpose

This project is a prototype for verifiable chain of custody of encrypted mobile evidence.

Security and abuse reports should be handled carefully. Do not disclose sensitive reports, illegal content, personal data, or exploitable vulnerabilities in public issues or pull requests.

## Do not submit sensitive material publicly

Do not submit any of the following in public GitHub issues, discussions, pull requests, demo environments, or test fixtures:

- illegal content,
- suspected illegal content,
- real evidence media,
- personal information,
- private keys,
- credentials,
- exact locations,
- exploit proof-of-concepts against live systems,
- screenshots containing sensitive data.

## Reporting security issues

Security issues should be reported privately to the maintainers.

A report may include:

- affected component,
- high-level impact,
- reproduction steps using synthetic data,
- expected behavior,
- actual behavior,
- suggested mitigation.

Avoid including real user data or real sensitive media.

## Reporting abuse concerns

Abuse concerns should be reported privately.

A report may include:

- snapshot ID or ledger ID,
- reason category,
- non-sensitive description,
- reporter contact method if appropriate.

Do not attach illegal, private, or sensitive content to public channels.

## Public disclosure

Please avoid public disclosure of exploitable vulnerabilities before maintainers have had a reasonable opportunity to investigate and mitigate.

## Prototype limitations

This prototype is not a production trust-and-safety system.

Because evidence content is end-to-end encrypted, the backend is not designed to inspect encrypted media. Abuse mitigation relies on product constraints, metadata-level reporting, limited distribution mechanics, and explicit acceptable-use boundaries.
